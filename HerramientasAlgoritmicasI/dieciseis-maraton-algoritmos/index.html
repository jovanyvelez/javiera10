🏅 La Rúbrica de Autoevaluación

⏱️ 45 minutos · Mide tu nivel con honestidad — sin puntos extras por mentir

📋 ¿Qué es una rúbrica de desempeño?

Una rúbrica es una tabla de criterios que se usan para evaluar el trabajo. Aquí vas a evaluarte a ti mismo con honestidad en 5 criterios. No hay nota numérica ni castigo: el objetivo es que tú sepas tu nivel real y qué áreas reforzar.

🎯 Los 5 criterios

✅ 1. Corrección
¿Tu código hace exactamente lo que pide el problema? ¿Los resultados son correctos?
⚡ 2. Eficiencia
¿Tu código hace el menor trabajo posible? ¿Evita recorrer arreglos más veces de lo necesario?
📖 3. Claridad
¿Un compañero de clase podría leer tu código y entender qué hace sin que le expliques?
✨ 4. Elegancia
¿Tu solución es simple y bonita? ¿Un programador senior la miraría y diría "esto está bien hecho"?
🏁 5. Completitud
¿Terminaste todos los retos de la maratón? ¿Pusiste atención hasta el final?

❓ Las 10 dudas más frecuentes

Una señal clara: el número de ciclos anidados. Si tienes un for dentro de otro for, tu algoritmo es O(n²): si tienes 100 datos, hace 10.000 operaciones. Si los datos crecen a 1.000, hace 1.000.000. Una solución más eficiente recorre el arreglo una sola vez usando un solo for. Mientras menos ciclos anidados, mejor.
Usa for cuando sabes CUÁNTAS veces repetir (ej: recorrer un arreglo de 10 elementos). Usa while cuando NO sabes cuántas veces (ej: pedir datos hasta que el usuario escriba "salir", o un juego hasta que el jugador gane). foreach es para recorrer colecciones sin preocuparte por el índice.
Porque lo creaste pero nunca le asignaste valores. En C#, int[] nums = new int[5] crea 5 espacios con valor 0, pero string[] nombres = new string[5] los crea con valor null. Siempre inicializa tus arreglos con valores reales: string[] nombres = { "Ana", "Luis" }, o asigna cada posición en un bucle.
Usa Console.WriteLine en puntos clave. Pon un print antes y después de la línea sospechosa para ver qué valores tienen tus variables. Si el valor antes es correcto y después no, el bug está en esa línea. Es la técnica más antigua y efectiva del mundo. Los programadores senior la usan todos los días.
Ctrl + C en la consola, o cerrar la ventana. Un ciclo infinito pasa cuando la condición de salida nunca se vuelve falsa. Casi siempre es: olvidaste el i++ en un for, o la variable del while nunca cambia. Revisa siempre: "¿mi contador se actualiza dentro del ciclo?"
Escribe en español ANTES de programar. Por ejemplo: "1. Crear un arreglo. 2. Recorrerlo con un for. 3. Si el número es par, sumarlo." Una vez que tienes los pasos en español, traducir a C# es fácil. Los programadores profesionales no escriben código de una: primero planean, después escriben.
El 90% de las veces es la condición de parada. ¿Usas < o <=? ¿El rango empieza en 0 o en 1? ¿Te pasaste un elemento? Truco: si quieres recorrer TODO el arreglo, usa i < arr.Length (sin =). Si quieres incluir el último, i <= arr.Length - 1.
Depende de qué comparas. Si comparas la misma variable contra varios valores literales (ej: opcion == "a", opcion == "b"...), usa switch (más limpio). Si las condiciones son complejas o diferentes entre sí (ej: edad > 18 Y compras > 5), usa if. La regla: si tienes más de 3 if sobre la misma variable, considera switch.
En bloques con comentarios. Un programa profesional se ve así: // 1. DECLARAR VARIABLES, // 2. LEER DATOS, // 3. PROCESAR, // 4. MOSTRAR RESULTADOS. Los comentarios no son para "tontos": son para que tu yo del futuro (y tus compañeros) entiendan el código. Un buen programa se lee casi como una receta de cocina.
Tres caminos posibles, todos válidos. (1) Algoritmos más avanzados: ordenamiento avanzado (Merge Sort, Quick Sort), búsquedas eficientes, recursividad. (2) Programación orientada a objetos: clases, objetos, herencia — la base de todo software moderno. (3) Desarrollo web: HTML, CSS, JavaScript — para crear páginas y apps reales. Cualquiera que elijas te espera con los brazos abiertos.
